# Prod & Dev Environments Design

**Date:** 2026-04-23
**Status:** Approved, ready for plan

## Goal

Split the single-environment SpinBike deployment on the runner machine into two isolated environments:

- **Prod** — `https://spinbike.newlevel.media`, fed by pushes to `main`. Contains real customer data.
- **Dev** — `https://spinbike-dev.newlevel.media`, fed by pushes to `dev`. Refreshed from prod nightly.

Both run on the same physical machine. CI must never touch prod when deploying dev, and vice versa.

## Why

Today the `deploy` CI job fires on every push to `dev` and installs the binary straight into `/home/newlevel/devel/spinbike/target/release/`, then restarts `spinbike.service`. That service serves `spinbike.newlevel.media` with the real customer DB. In effect, **`dev` branch deploys to prod**. This is unsafe as the app enters production use: a broken build on `dev` can corrupt the real DB or take the site down until a fix lands.

Moving to two environments gives us a safe target (`spinbike-dev.*`) for `dev`-branch work and reserves prod for intentional `main` pushes.

## Non-goals

- Containerisation (systemd + file-per-env is sufficient; added complexity not justified)
- Separate staging env distinct from dev (dev *is* the staging env for this one-person team)
- Zero-downtime or blue/green deploy (2–3 s systemd restart is acceptable for this workload)
- External secrets manager (machine-local env files under root-owned `/etc/default/` are adequate)
- Rollback automation (forward-fix via revert commit; DB backups cover data corruption)

## Architecture

### File & process layout on the runner

```
/opt/spinbike/
├── prod/
│   ├── spinbike-server              # binary, atomically replaced per deploy
│   ├── spinbike.db                  # prod SQLite DB (migrated from existing
│   │                                # /home/newlevel/devel/spinbike/spinbike.db)
│   └── backups/
│       └── spinbike-YYYYMMDD-HHMMSS.db  # rolling last-10
└── dev/
    ├── spinbike-server
    └── spinbike-dev.db              # seeded empty; nightly sync from prod

/etc/systemd/system/
├── spinbike.service                 # PORT=8080, prod paths, EnvironmentFile=/etc/default/spinbike-prod
├── spinbike-dev.service             # PORT=8081, dev  paths, EnvironmentFile=/etc/default/spinbike-dev
├── spinbike-sync-dev.service        # oneshot: cp prod.db → dev.db atomically + restart dev
└── spinbike-sync-dev.timer          # OnCalendar=daily 03:00

/etc/default/
├── spinbike-prod                    # JWT_SECRET, CORS_ORIGIN (0640 root:newlevel)
└── spinbike-dev                     # same, different secret
```

Prod and dev binaries live in distinct directories so a dev deploy's `install -Dm755` cannot overwrite the inode prod is running from. Existing CI already uses `install` for the same reason (avoids ETXTBSY on mmap'd binary); we keep the pattern per env.

### Reverse proxy

Existing Cloudflare tunnel `4093c494-b31d-4eb7-8fcb-6c5948f5d4b2` hosts both hostnames. `~/.cloudflared/config.yml`:

```yaml
tunnel: 4093c494-b31d-4eb7-8fcb-6c5948f5d4b2
credentials-file: /home/newlevel/.cloudflared/4093c494-b31d-4eb7-8fcb-6c5948f5d4b2.json
ingress:
  - hostname: spinbike.newlevel.media
    service: http://localhost:8080
  - hostname: spinbike-dev.newlevel.media
    service: http://localhost:8081
  - service: http_status:404
```

One-time manual setup steps (require Cloudflare auth; must not live in CI):

```bash
cloudflared tunnel route dns spinbike spinbike-dev.newlevel.media
sudo systemctl restart spinbike-tunnel.service
```

### Systemd units

`spinbike.service` becomes:

```ini
[Unit]
Description=SpinBike PWA Server (prod)
After=network.target

[Service]
Type=simple
User=newlevel
Environment=PORT=8080
Environment=DATABASE_PATH=/opt/spinbike/prod/spinbike.db
EnvironmentFile=/etc/default/spinbike-prod
ExecStart=/opt/spinbike/prod/spinbike-server
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

`spinbike-dev.service` is the same shape with `/opt/spinbike/dev/` paths, `PORT=8081`, `EnvironmentFile=/etc/default/spinbike-dev`.

`/etc/default/spinbike-prod` (0640, root:newlevel):

```
JWT_SECRET=<64-hex generated by openssl rand -hex 32>
CORS_ORIGIN=https://spinbike.newlevel.media
```

`/etc/default/spinbike-dev` is the same shape with a *different* secret and `CORS_ORIGIN=https://spinbike-dev.newlevel.media`.

This replaces today's inline `Environment=JWT_SECRET=demo-secret-key`, which was insecure and committed to `/etc/systemd`.

### Nightly prod→dev sync

`spinbike-sync-dev.service` (Type=oneshot):

```
ExecStart=/usr/bin/cp /opt/spinbike/prod/spinbike.db /opt/spinbike/dev/spinbike-dev.db.tmp
ExecStart=/usr/bin/mv -f /opt/spinbike/dev/spinbike-dev.db.tmp /opt/spinbike/dev/spinbike-dev.db
ExecStart=/usr/bin/systemctl restart spinbike-dev.service
```

The tmp+atomic-mv avoids the dev server reading a half-copied file. The restart ensures sqlx reopens the DB with the new inode.

`spinbike-sync-dev.timer`:

```ini
[Timer]
OnCalendar=*-*-* 03:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

### CI deploy split

Today's single `deploy` job becomes two mutually-exclusive jobs in `.github/workflows/ci.yml`:

```yaml
deploy-dev:
  if: github.event_name == 'push' && github.ref == 'refs/heads/dev'
  runs-on: [self-hosted, spinbike-deploy]
  needs: [test, build-wasm, e2e]
  steps:
    - checkout
    - trunk build --release (spinbike-ui)
    - touch crates/spinbike-server/src/routes/static_files.rs   # force rust-embed rebake
    - cargo build --release --bin spinbike-server
    - install -Dm755 target/release/spinbike-server /opt/spinbike/dev/spinbike-server
    - sudo -n systemctl restart spinbike-dev.service
    - wait-for-health https://spinbike-dev.newlevel.media (30 tries × 2 s)
    - Playwright @smoke against SMOKE_BASE_URL=https://spinbike-dev.newlevel.media

deploy-prod:
  if: github.event_name == 'push' && github.ref == 'refs/heads/main'
  runs-on: [self-hosted, spinbike-deploy]
  needs: [test, build-wasm, e2e]
  steps:
    - checkout
    - Backup: cp /opt/spinbike/prod/spinbike.db
             /opt/spinbike/prod/backups/spinbike-$(date +%Y%m%d-%H%M%S).db
             && ls -t /opt/spinbike/prod/backups/spinbike-*.db | tail -n +11 | xargs -r rm
    - trunk build --release (spinbike-ui)
    - touch crates/spinbike-server/src/routes/static_files.rs
    - cargo build --release --bin spinbike-server
    - install -Dm755 target/release/spinbike-server /opt/spinbike/prod/spinbike-server
    - sudo -n systemctl restart spinbike.service
    - wait-for-health https://spinbike.newlevel.media
    - Playwright @smoke against SMOKE_BASE_URL=https://spinbike.newlevel.media
```

Trigger config:
- `on.push.branches: [main, dev]` stays (both branches need CI).
- `on.pull_request.branches: [main]` stays (PRs run CI, skip both deploy jobs).
- The `if:` guards on each deploy job are the only branch→env routing.

**Sudoers:** the runner user currently has `NOPASSWD` sudo for `systemctl restart spinbike.service`. Extend to also cover `spinbike-dev.service`:

```
newlevel ALL=(root) NOPASSWD: /bin/systemctl restart spinbike.service, /bin/systemctl restart spinbike-dev.service
```

### Existing data migration

At rollout, the current `/home/newlevel/devel/spinbike/spinbike.db` moves to `/opt/spinbike/prod/spinbike.db`. Sequence, performed manually once:

```bash
# Stop current prod
sudo systemctl stop spinbike.service

# Create env dirs
sudo install -d -o newlevel -g newlevel /opt/spinbike/prod /opt/spinbike/prod/backups /opt/spinbike/dev

# Move prod DB (cp, not mv — original stays as offline backup for ~2 weeks)
sudo cp /home/newlevel/devel/spinbike/spinbike.db /opt/spinbike/prod/spinbike.db
sudo chown newlevel:newlevel /opt/spinbike/prod/spinbike.db

# Seed dev DB by copying prod (first-time equivalent of nightly sync)
sudo cp /opt/spinbike/prod/spinbike.db /opt/spinbike/dev/spinbike-dev.db
sudo chown newlevel:newlevel /opt/spinbike/dev/spinbike-dev.db

# Install current binary into BOTH env dirs so both services can start
# before first CI deploy. CI will overwrite each as pushes happen.
sudo install -Dm755 /home/newlevel/devel/spinbike/target/release/spinbike-server \
    /opt/spinbike/prod/spinbike-server
sudo install -Dm755 /home/newlevel/devel/spinbike/target/release/spinbike-server \
    /opt/spinbike/dev/spinbike-server

# Write /etc/default/spinbike-prod and /etc/default/spinbike-dev (generate
# JWT_SECRET with `openssl rand -hex 32`, one unique value per env).
# Install new unit files for spinbike.service, spinbike-dev.service,
# spinbike-sync-dev.service, spinbike-sync-dev.timer.

sudo systemctl daemon-reload
sudo systemctl start spinbike.service spinbike-dev.service
sudo systemctl enable --now spinbike-sync-dev.timer

# Cloudflare tunnel changes (require Cloudflare auth on this machine)
cloudflared tunnel route dns spinbike spinbike-dev.newlevel.media
# Edit ~/.cloudflared/config.yml per "Reverse proxy" section above
sudo systemctl restart spinbike-tunnel.service
```

Ordering note: Cloudflare DNS + tunnel config must be in place before the post-deploy smoke-test step of the first CI run can reach `https://spinbike-dev.newlevel.media`. The one-time script above handles this.

The existing `/home/newlevel/devel/spinbike/spinbike.db` stays in place (read-only from the server's perspective) as a belt-and-braces offline backup. After 1–2 weeks of stable operation it can be deleted.

## Data flow

**Prod deploy (happy path):**
1. PR from `dev` → `main` merges → push event fires.
2. CI runs lint, test, e2e, build-wasm (all on ubuntu-latest).
3. `deploy-prod` gated `if:` matches → runs on self-hosted runner.
4. DB backup step copies `/opt/spinbike/prod/spinbike.db` → `backups/spinbike-<ts>.db`, prunes to 10 most recent.
5. `cargo build --release` produces fresh binary.
6. `install -Dm755` replaces binary.
7. `systemctl restart spinbike.service` → server starts, runs `db::run_migrations` against prod DB (idempotent; only new migrations apply).
8. Health-check loop polls `https://spinbike.newlevel.media` for up to 60 s.
9. Playwright `@smoke` suite runs against prod URL (read-only tests).

**Dev deploy (happy path):**
Same as prod, minus the DB backup step. Targets `spinbike-dev.service` / `spinbike-dev.newlevel.media` / port 8081.

**Nightly sync:**
03:00 local time: timer fires → oneshot service copies prod DB → dev DB → restarts dev service. Dev DB is now a same-schema, same-data snapshot of prod as of 03:00. Any dev-only test data written during the day is wiped.

## Error handling

**Deploy step fails before binary install:** service keeps running old binary. No user impact. CI job marked failed; fix and push again.

**Deploy step fails after binary install, before health check:** systemd `Restart=always` keeps the new binary running. If it crash-loops, the runner user investigates: `journalctl -u spinbike.service -n 100`. Manual rollback if needed: `install` the previous binary back and `systemctl restart`.

**Health check fails:** CI job reports failure with logs from `journalctl -u <service> --no-pager`. Service is left as-is; manual recovery required.

**Migration corrupts prod data:** restore from `/opt/spinbike/prod/backups/spinbike-<ts>.db` — the snapshot taken *before* this deploy:
```bash
sudo systemctl stop spinbike.service
sudo cp /opt/spinbike/prod/backups/spinbike-<ts>.db /opt/spinbike/prod/spinbike.db
sudo systemctl start spinbike.service
```

**Nightly sync fails:** timer logs to journal; dev DB is last-good from previous successful sync. No impact on prod. Alert is surfaced the next time a developer checks the dev env.

**CORS mismatch:** if `CORS_ORIGIN` in `/etc/default/spinbike-*` doesn't match the hostname the user is hitting, the server rejects browser requests with a CORS error visible in browser console. Detected by smoke test (Playwright requires a working session cookie).

## Testing

**Unit tests:** none of this change modifies Rust code — it's systemd units, shell, YAML. CI itself is the test.

**E2E:** existing E2E suite unaffected. Continues to run against ephemeral `/tmp/spinbike-e2e.db` in CI before either deploy job.

**Smoke tests (post-deploy):** today's existing `@smoke` suite runs against both envs, just with a different `SMOKE_BASE_URL`. No new tests added here — the existing suite plus the `wait-for-health` step is sufficient to catch the obvious "site is down" and "login form missing" regressions. Write-path smokes are intentionally avoided against prod.

**Manual verification after first deploy:**
1. `curl -I https://spinbike.newlevel.media` → 200
2. `curl -I https://spinbike-dev.newlevel.media` → 200
3. Both serve *different* server versions when VERSION differs between branches (confirms no cross-contamination)
4. Log into prod with real user, verify card data intact
5. Log into dev with same user, verify same data visible (nightly sync worked)

## Rollout risk

**Highest-risk step: the one-time data migration.** Moving `spinbike.db` and changing its path + service config is the only step that touches real customer data. Mitigations:
1. Performed manually, not by CI
2. `cp` preserves original file — original stays on disk until explicitly deleted
3. Done during a known-quiet window (fitness centre off-hours)
4. Roll back by editing the service unit back to the old path + `systemctl restart` — takes ~30 s

**Second-highest: sudoers permission error.** If the extended sudoers line has a syntax error, `sudo -n systemctl restart` fails for the CI user and deploys stop working. Mitigation: `visudo -c` the file before saving; test with `sudo -n systemctl restart spinbike-dev.service` from the runner user before merging CI changes.

**Lowest: DNS/tunnel.** Cloudflare DNS propagation is fast (<30 s); if the new hostname doesn't resolve, only the dev env is affected. Prod keeps running untouched.

**Modified:**
- `.github/workflows/ci.yml` — `deploy` job splits into `deploy-dev` + `deploy-prod`
- `/etc/systemd/system/spinbike.service` — paths change to `/opt/spinbike/prod/`, secrets move to `EnvironmentFile`

- `/etc/sudoers.d/spinbike` — add dev service to NOPASSWD list
- Cloudflare DNS — new CNAME `spinbike-dev.newlevel.media`

**Unchanged:**

- Existing E2E tests.
- Branch workflow (dev → PR → main remains).
